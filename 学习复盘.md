## 一、向量化（Embedding）完整原理

这是整个 RAG 系统最核心的环节——把人类语言变成机器能计算的数字序列。

### 1.1 整体流程

```
"我喜欢猫"
  → 分词 → 查词表得初始向量
  → 经过 12 层 Transformer Block
  → 取最后一层所有词向量做平均（池化）
  → 得到 512 维句向量
```

### 1.2 分词与初始向量

```
"我喜欢猫" → ["我", "喜欢", "猫"] → 查词汇表 → [1057, 3821, 892]
                                           ↓ 查词嵌入矩阵
                          "我"   → [0.10, -0.20, 0.30, 0.05]
                          "喜欢" → [0.40,  0.15,-0.25, 0.08]
                          "猫"   → [-0.30, 0.22, 0.12, 0.03]
```

> 示例用 4 维便于演示，真实 BGE 模型输出 512 维。

### 1.3 Self-Attention（自注意力）—— 让每个词"理解上下文"

**核心问题：** "猫"的初始向量永远一样，但"我家养了一只猫"和"电脑内存只有 8G 太猫了"中"猫"含义完全不同。Self-Attention 让每个词根据上下文重新组装自己。

**QKV 三把钥匙：**

| 钥匙 | 全称 | 含义 | 类比 |
|------|------|------|------|
| Q | Query | "我在找什么？" | 求职者的技能需求 |
| K | Key | "我有什么？" | 简历上的标签 |
| V | Value | "我的干货？" | 面试的实际内容 |

每个词 × 三个权重矩阵 = Q、K、V 三个向量：

```
x₁ × W_Q = Q₁      x₁ × W_K = K₁      x₁ × W_V = V₁
```

**W_Q、W_K、W_V 是模型训练出来的，初始为随机数，通过海量文本的反向传播自动调优。**

**计算过程：**

```
Step 1: Q × K → 注意力分数矩阵
  Q₁·K₁  Q₁·K₂  Q₁·K₃
  Q₂·K₁  Q₂·K₂  Q₂·K₃    （分数越大 = 两个词越相关）
  Q₃·K₁  Q₃·K₂  Q₃·K₃

Step 2: Softmax 逐行归一化 → 0~1 的权重（每行总和为 1）

Step 3: 权重 × V → 融合后的新向量
  "猫"的新向量 = 0.33×V₁ + 0.33×V₂ + 0.34×V₃
  此时"猫"已经吸收了"喜欢"和"我"的语义
```

### 1.4 Multi-Head Attention（多头注意力）

模型同时做 8 次 Self-Attention，每组 W_Q/W_K/W_V 不同，关注不同的语义模式：

```
Head 1：语法结构  Head 2：情感色彩  Head 3：实体关系  ...  Head 8：共现模式
        ↓ 8 个头结果拼接（8×4=32维）
        ↓ × W_O(32×4)（融合多头信息，W_O 同样是训练出来的）
        ↓ 输出 4 维
```

### 1.5 Transformer Block 完整结构

一层 Transformer Block 不只有 Attention，还有两个重要步骤：

```
输入 → Self-Attention → Add & Norm → Feed-Forward → Add & Norm → 输出
```

- **残差连接（Add）：** `output = input + attention_output`，防止深层网络退化。类比：听了别人意见后的最终观点 = 原始观点 + 调整量，而非完全替换
- **LayerNorm：** 把数值拉回标准范围，防止越跑越偏
- **Feed-Forward：** `ReLU(xW₁+b₁)W₂+b₂`，增加非线性变换能力

12 层叠加后，各层向量变化：

```
        初始              第1层后            第6层后            第12层后
"我"    [0.10,-0.20,...]  [0.025,0.018,...] [0.017,0.019,...] [0.014,0.016,...]
"喜欢"  [0.40,0.15,...]   [0.018,0.022,...] [0.015,0.021,...] [0.013,0.018,...]
"猫"    [-0.30,0.22,...]  [0.008,0.017,...] [0.012,0.020,...] [0.015,0.020,...]
         ↑ 各不相关         ↑ 开始靠拢         ↑ 深度融合         ↑ 融为一体
```

### 1.6 池化 → 句向量

第 12 层三个词向量取平均：

```
[0.014,0.016,...] + [0.013,0.018,...] + [0.015,0.020,...] ÷ 3
= [0.014, 0.018, 0.014, ..., 0.015]  ← 512 维句向量
```

**语义相近的句子 → 这 512 个数字的几何距离也近 → 可做相似度搜索。**

### 1.7 为什么是 512 维？

- BGE-small 的隐藏层宽度就是 512，不是随便挑的
- 神经网络维度通常是 2 的幂次（64→128→256→512→768→1024→1536），GPU 计算效率最高
- 这是效果与成本的权衡：维度越高越精准，但越慢越占空间
- 不同模型维度不同：BGE-base=768，BGE-large=1024，OpenAI=1536

## 二、当前系统的检索层

```
用户提问 "RAG 有什么优点？"
    │
    ▼
BGE Embedding → 512 维向量         ← 语义层面：把文字变成"意思向量"
    │
    ▼
ChromaDB 向量数据库                 ← 存储目录: ./chroma_db/
    │  排序算法: 余弦相似度
    │  索引加速: HNSW（层次可导航小世界图）
    ▼
Top-K=3 最相关文档块 → 拼接 Prompt → LLM 生成回答
```

### 2.1 HNSW 索引是什么

没有索引时，每次搜索要把问题向量与所有文档向量逐一比对（暴力搜索）。HNSW 构建一张分层图：

```
第2层（稀疏）:  节点 A ──────────── 节点 F ──────────── 节点 K
                  │                                       │
第1层（中等）:  节点 A ── 节点 C ── 节点 F ── 节点 H ── 节点 K
                  │          │                  │
第0层（密集）:  节点 A─B─C─D─E─F─G─H─I─J─K       ← 全量节点
```

搜索时从高层稀疏层快速定位到附近，再逐层下降精确查找。类比：**高速公路 → 国道 → 乡道**。

HNSW 只是加速工具，它和检索策略（语义/关键词/混合）无关——无论哪种检索策略，都可以用 HNSW 加速，换的是评分公式，不是索引结构。

## 三、三种检索策略对比

| 策略 | 方式 | 优势 | 劣势 |
|------|------|------|------|
| **语义检索** | Embedding 转向量 → 余弦相似度 | 理解同义词、上下文 | 对编号、专有名词不够敏感 |
| **关键词检索** | BM25 算法（词频 × 逆文档频率） | 精确匹配术语 | 不理解同义词 |
| **混合检索** | 语义 + 关键词结果合并去重 | 覆盖面更广 | 可能引入噪音 |

### 3.1 BM25 原理

某个词在一篇文档中出现次数越多 → 分数越高（词频 TF）。但"的"/"了"这类词在所有文档都有 → 降低权重（逆文档频率 IDF）。BM25 = TF × IDF。

### 3.2 使用场景判断

```
精确查询（编号、术语）    → 关键词 > 混合 > 语义
模糊查询（日常问题）      → 语义 > 混合 > 关键词
复杂查询（精确+语义）     → 混合 > 语义 > 关键词
```

**没有银弹（No Silver Bullet）** ——这个概念来自计算机经典论文，比喻不存在一种能完美解决所有问题的方案。

## 四、表格数据处理踩坑与修复

### 4.1 问题背景

宿舍分配 Excel 文件：55 行真实数据（每行一个房间，含 4 个学生），但系统上传后出现三大问题：

| 问题 | 现象 | 根因 |
|------|------|------|
| 1 | 55 行变 1128 行 | 空行未过滤（Excel 有格式残留在 1129 行） |
| 2 | 搜"卿毅在 4020"失败 | 所有行拼成一个大文本再切块，行被拦腰斩断 |
| 3 | 表头全是 None | 合并单元格导致 openpyxl 只读到第一个格子有值 |

### 4.2 修复方案

**修复 1：空行过滤**

```python
rows = [r for r in rows if any(v.strip() for v in r)]
```

**修复 2：每行一个独立 Document**

```
改前：                         改后：
┌──────────────────────┐       ┌── 摘要
│ 摘要 + 所有行拼一起    │       ├── 第1行: 序号:161 | 公寓:...
│                      │       ├── 第2行: 序号:162 | ...
│ （55行）              │       ├── 第3行: ...
└──────┬───────────────┘       └── 第54行: ...
       ↓ split_documents             ↓ split_documents
  被切块，行会断裂              每行 < chunk_size，不会被切
```

**修复 3：合并单元格处理 `_clean_table`**

```python
# Step 1: 前向填充 —— None 继承上一个有效表头
学号, None, None, None, 姓名 → 学号, 学号, 学号, 学号, 姓名

# Step 2: 合并连续同名列
学号, 姓名 → 学号: 20230001 CS 3班 001 | 姓名: 张三
```

### 4.3 重构：消除 CSV/XLSX 的重复代码

发现 `load_csv` 和 `load_xlsx` 的转换逻辑完全相同（只有读文件方式不同），抽出共用函数 `_rows_to_documents`：

```python
# 私有函数（下划线开头）—— 只有本文件内部调用
def _rows_to_documents(rows, headers, filepath):
    """表格转为每行一个 Document"""
    ...

# 公开函数 —— upload.py 调用
def load_csv(filepath):
    rows = csv_reader()       # 读文件
    return _rows_to_documents(rows, headers, filepath)

def load_xlsx(filepath):
    rows = openpyxl_read()    # 读文件
    return _rows_to_documents(rows, headers, filepath)
```

**何时该抽共用逻辑？** 同一个逻辑出现第二次时就该抽。只在第一次出现时是"过度设计"。公开函数管"做什么"，私有函数管"怎么做"。

```
D:\Pythoncode\RAG\
├── .gitignore              ← 排除 venv/ chroma_db/ __pycache__
├── doc_loader.py           ← 文档加载（TXT/PDF/CSV/XLSX）
```

